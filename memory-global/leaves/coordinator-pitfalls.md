# Typical coordinator pitfalls

## Anti-patterns

| Symptom | Better |
|---|---|
| Root does most edits via Bash/Edit/Write | `Task → developer` for code; invoke `overcome-difficulty` skill when stuck |
| Extend existing pattern (keyword scoring, regex routing, hand-coded heuristics) without checking it still fits — especially when an LLM is now in the pipeline reading the same data | Audit pattern fit before extending. In LLM-mediated pipelines, prefer natural-language prefs and let the LLM filter semantically — keyword lists are usually pre-LLM holdovers. |
| Fabricate "reasonable defaults" in a user-facing artifact (config field, description, padded list, helpful-sounding example) the user never requested | Include only what was stated or strictly required. Empty / ask / minimum and flag the gap — never silently extrapolate user preferences. |
| Full pipeline rerun to debug one stage | Minimal retest; read project memory runbook before re-launching |
| Second CLI binary for one-off experiment | Local script/stash; one entry point in the repo |
| User feedback on process → only apology | Invoke `self-improvement` skill in the same turn |
| User asks "why no self-improvement?" / confirms "run it" | Invoke `self-improvement` skill in the same turn — the reminder is feedback |
| Conditional "if not done" + artifact missing in repo → redo without status check | Read org runbook first; closed/done may mean stop — ask the user |
| VCS branch/commit on a scoped task without user ask | Read-only + report until the user confirms scope |
| Unknown internal term → guess in code | Infra-consultant subagent if present in `~/.claude-agent/agents/`, else intrasearch / domain MCP |
| Domain runbook pasted into a generic agent prompt or `CLAUDE.md` | Memory leaf (global or project) and link in the plan |
| Substantive task confirmed-resolved after the first sub-stage only (plan written → "task resolved?") | Re-read the original user message. Resolution = X done, not plan-X done. Ask "first sub-stage accepted?" and continue execution on confirmation. |
| Misapply "don't fabricate" by **deleting** the field instead of asking for a source | "Don't fabricate" includes "ask or find a source". Removing a required field (estimate, threshold, deadline) is a regression — when the plan format requires it, `CLARIFY:` / ask user instead. |
| Proceed on an action that depends on a file after a system-reminder said the file changed, without re-reading it | When a `<system-reminder>` reports a file was modified — **Read it before any next action that depends on its prior contents**. Skipping cost a full spawn budget on one occasion (developer child accumulated 48 permission_denials at $3.05 because reverted `settings.local.json` was assumed still patched). |
| Launch a long-running spawn (`claude -p`, especially `kind=developer` with `--permission-mode bypassPermissions`) and not check on it for tens of minutes | Periodically inspect the spawn's transcript while it runs (rule of thumb: every ~5 min for developer spawns). The transcript jsonl is the freshest `.jsonl` under `~/.claude/projects/<sanitized-cwd>/` (find by mtime). Look for: wrong `cwd`, writes/commits outside the assigned mount, `ya package`/`docker push`/smoke-tests of unrelated tickets. **Kill early** if the spawn diverges; reusing the budget on rescoping a runaway is cheaper than waiting for it to exhaust. Observed cost: ~$3 burned with zero on-target artifacts when developer cd'd into a "reference" mount and started building there. **Kill the subtree, not the wrapper pid** — `python3 scripts/kill-tree.py <pid>` (or `kill -- -<pgid>`) reaps the whole process group; a bare `kill <pid>` orphans the `claude -p` children, which keep running and burning budget. Caveat: do **not** reap the intentionally-detached `nohup … &` pollers from `long-job-monitoring.md` this way — they are designed to survive the session. |
| Use `--permission-mode bypassPermissions` as a blanket "trusted local" grant without write-scope constraint in the prompt | bypassPermissions removes all harness write barriers, so the child can `cd` and `arc commit` anywhere the kernel lets it. The dossier/prompt must contain explicit hard-deny list ("no cd / no Write / no arc commit outside `<mount>`") plus a self-check at session start (`pwd` ⊆ expected mount). Without that the child treats reference mounts as fair game for "understanding through execution". |
| After killing a spawn for off-scope behavior, check `arc log` / `git log` on the assigned branch, not just `arc status` / `git status` | `status` only shows uncommitted changes; a spawn that committed on-scope work and then drifted to off-scope work shows clean status but has a new commit. Skipping this cost a redundant $3.80 spawn that re-verified work the prior killed session had already committed. After kill: `arc log -n 5 --oneline` (or git equiv) on the assigned branch is mandatory before deciding the next move. |
| Wrote project-file paths into a plan assuming `<project>/.claude/` is the tracked target, when it's an arc-ignored symlink to shared storage | Before planning edits to project files (config / hooks / settings / memory): `ls -ld .claude` + check `.arcignore`, and read the project's memory index — the real target may be a shared-storage dir the `.claude/` symlink points into. Cost of skipping: a wasted `developer` spawn (~$1.74) + a re-approval cycle when the spawn returned `REPLAN:` (2026-06-22). |
| Advanced to the next plan stage (or reported the task complete) without comparing the actual outcome to the stage's `Expected result image:` | CLAUDE.md § Coordination spine (driven by `agentctl`) — verification has two mandatory layers: after each stage against its result image, and after the last stage against the plan's `## Final verification`. On mismatch → `overcome-difficulty`. Treat a skipped check as a difficulty itself: stop and run it before advancing. |
| Treated bare gratitude (`thanks` / `спасибо` / `perfect` alone) as resolution confirmation, or waited for user gratitude before asking | CLAUDE.md § On task resolution — close the loop **proactively** the moment Final verification passes (recap + explicit ask), do not wait for gratitude. When gratitude comes, treat it as a prompt to ask, not as the answer. `scripts/hook-resolution-reminder.py` (UserPromptSubmit) emits a nudge on brief gratitude as a safety net. |
| Confirmation / apply-skip / push / resolution question asked as free text ("Применять?", "Так и оставим?", "Запушить?", "Считаем резолвнутой?") at end of turn | CLAUDE.md § Escalation to the user — `AskUserQuestion` is **mandatory** whenever the answer is binary or one-of-N you can list. "Apply this?" is not open-ended; it's exactly what the structured UI exists for. Bundle multiple end-of-turn binary decisions into a single `AskUserQuestion` call rather than splitting structured asks + free-text sign-off across the turn. Failure mode: user has to type "да" where one click would suffice, and the closing protocol's structured-confirm affordance is lost. |
| Asked the user for permission on a side-effect-free action (Read / Grep / WebFetch / `--help` / `--dry-run` / MCP `get_*` `list_*` `search_*`) or on an action explicitly declared in the approved plan's scope | CLAUDE.md § Acting without asking — side-effect-free classes and plan-scope-declared actions are pre-authorized. Re-asking erodes the "structured confirm = something I actually need to decide" affordance and wastes the user's time. Carve-outs and full class list in `memory-global/leaves/acting-without-asking.md`. |
| Burned 3+ tool calls "to be sure" about a new MCP/CLI's side-effect class | Budget is **1 lookup** (`--help` / `ToolSearch` / `Read` SKILL.md). If still unclear → `PERMISSION-REQUEST:` immediately. One click for the user beats minutes of agent thrashing. |
| Declared a capability unavailable and handed the task back to the user ("token's broken — you post it", "no access to X, do it yourself") after a **single** documented method failed | "My one method failed" ≠ "impossible" — it means "I haven't found the working method yet." Exhaust alternative sources before offloading: token files (`ls ~/.*token*`), env vars, an alternate skill/CLI, the project memory leaf. **Direct evidence the action is doable — a leaf's `verified by`, a past successful run, the user saying "you did this before" — proves a working method exists; find it, don't hand the task back.** Offloading to the user is the last resort, not the second step. (Cost: stalled a ready tracker comment + wrote a wrong "user posts" default into memory, when `~/.tracker-token` held a working `startrek:write` token used in prior sessions.) |
| Quietly expanded the approved plan's scope mid-execution ("while I was at it I fixed X in the adjacent file") | CLAUDE.md § Acting without asking — silent scope expansion is a substantive plan change, not a refinement. Stop, state the diff (was → now → why), confirm via `AskUserQuestion`. Adding a file to `Reference files` is substantive even if the added work feels obviously beneficial. |
| Single hypothesis anchored on loudest symptom (timeout, OOM, cancel, "model never started") without reference run or graph topology | Before infra logs: compare failing vs known-good run at **block order**; map Stop/Start/shared-WI causality; `arc diff` on ticket paths. Read `workflow-debug-investigation.md`; invoke `overcome-difficulty` with ≥2 falsifiable hypotheses. |
| Child WI `cancel` / timeout read as infra failure without parent graph context | Read terminal status only in context of parent block order — baseline may show the same cancel as normal cleanup. Topology pass before YT/Nirvana deep dive. |
| Same root-cause explanation across 2+ investigation iterations with no new evidence | Rewrite declaration (Expected/Actual/Mismatch); full inline OD cycle; in Cursor after 2 non-converging cycles → `~/claude-agent-instructions/scripts/spawn-cursor-escape.py` (fresh `agent -p`), then structured escalate to the user if still stuck — not another relaunch. |
| Investigated a difficulty into a constrained menu / "near-intractable" framed around a **mechanism** ("API can't pin the path", "X can't cross the boundary") and relayed it to the user, when a simpler primitive removes the **underlying** difficulty | Before declaring intractable or relaying a menu: restate the **functional ground** (the real desired-vs-actual — "real data at a controlled path", not "vh3-native output pinned to path"), then enumerate the **simplest primitive** that achieves it — especially one already demonstrated in code/artifacts you've read (DEEPAGENT-430: `finalize` already did `yt.copy(..., force=True)` to a chosen path; train already reads fixed option-paths). A specialist returning "intractable + menu" is a signal to re-derive the ground **yourself**, not to relay. Cost: two developer spawns + a user round-trip to rediscover an explicit `yt move` to a fixed option-path. |
| Surfaced a value/parameter choice to the user (`AskUserQuestion`) premised on a half-remembered constraint, when the answer was already determinable from readable artifacts — in-code preset/config comments, sibling tickets + reviewer comments, prior successful-run launchers/logs | Before asking the user to choose a config/risk parameter, exhaust readable evidence and **verify the constraint you'd cite**: grep the preset/config + its comments, read sibling tickets + reviewer comments, check prior successful-run launchers/logs. Ask only genuine unknowns; record the resolved fact as a memory leaf so it isn't re-derived. *(DEEPAGENT-430: asked "does 131k context OOM?" when `_PROD_PRESET`'s own comment + DEEPAGENT-403 green 16-GPU runs + DEEPAGENT-429/Ruslan already answered it — the OOM was 8-GPU-only.)* |
| Anchored target-artifact identification on `cwd` — assumed an ambiguous "README" / "architecture" / "the docs" meant the file under the current project, and treated the user's contradiction-question ("why does the *global* X know about project Y?") as something to defend rather than a signal to revise the hypothesis | **To hit the artifact the user actually means:** when a reference is ambiguous about scope (root vs project, which repo / which file), resolve it **before** acting — name your assumed target back to the user, or `AskUserQuestion` root-vs-project. A user question pointing out an inconsistency in your framing is **disconfirming evidence** — switch hypotheses, don't justify the current one. (Cost: ~6 turns misdirected at the wrong README before an explicit correction.) |
| Enumerated variants/options for a design/architecture question from the familiar **passive-mechanism catalog only** — listing known mechanisms (precedence lattices, governance gates, merge strategies) but omitting **active-agent** variants (a dedicated agent that synthesizes and proposes ranked resolutions) and the **cadence axis** (batch-accumulation vs per-item processing), and not deriving options from the system's own **difficulty → generalization → invariant** ontology | **To present a complete option space:** before listing variants for a design/architecture question, span orthogonal axes — passive-mechanism vs active-agent, batch vs continuous, mechanical-precedence vs synthesized-invariant. Ground each option in the difficulty it removes, and ask whether an **induced invariant subsumes** competing options (dialectical synthesis) rather than only cataloguing known mechanisms. The system's functional-ground ontology is itself an option-generator, not just a justification check. (Cost: user had to correct an incomplete variant set on the consensus-architecture research task, 2026-06-26.) |
| Made an existence / absence / **completeness** claim about the agent system ("there's no X skill/agent", "the system has these specializations", "the description is/isn't complete") from the **global layer alone** (`~/.claude-agent/CLAUDE.md` + global memory), ignoring the **project-local `.claude` layer** active in cwd | **To make a correct claim about what the system contains, enumerate ALL active config layers** — the running system is their **merge**: global instructions/memory **plus** the project-local `.claude/` in (or above) cwd, with its `agents/`, `skills/`, `agent-memory/`, `rules/`. cwd is often a **project mount**, not a bare dir. Before any inventory / "does X exist" / completeness claim, `ls <cwd>/.claude` and walk up to the project root (e.g. the deepagent project's `agents/` + `skills/`). A single-layer view yields false absence claims. (Cost: claimed the "Яндекс-Гуру" agent didn't exist while critiquing a system-description wiki page; it is a real project agent at `…/deepagent/agents/yandex-guru.md`, 2026-06-29.) |
| Doubted a user's stated requirement (asked a clarifying question premised on "X doesn't exist") when the apparent contradiction was really the agent's **stale local snapshot** — treated an unpulled local repo state as ground truth | CLAUDE.md § Escalation to the user — suspect your own freshness first: `pull` / `fetch` / re-read the authoritative source (fresh state may live on another branch/machine) before doubting the requirement. Evaluate requirements for adequacy AND non-contradiction, but resolve a perceived contradiction to root — self-staleness included — before escalating. (Cost: doubted the correct `claude-agent` requirement built on a stale local Core checkout; the command existed on the unpulled `origin/isolate-agent-config-root`, 2026-07-01.) |
| Presented a landscape/survey ("find the systems", "what exists for X") as complete when the research subagent's primary search modality (wiki / intrasearch) had **failed** and it silently degraded to a single modality (code search), reporting the coverage gap only as a trailing caveat — which the coordinator accepted as-is | A survey's done criterion is **universally quantified** (CLAUDE.md § cognition: never established by the instances you found) — completeness requires the multi-modal sweep to have actually **run** (wiki + code + tracker + web, each blind to the others' misses: wiki-first systems have little code footprint and vice versa). A subagent's failed modality **invalidates the completeness claim, it doesn't annotate it**: (a) retry through an alternate channel (direct MCP tools vs curl, a different agent, WebFetch); (b) if the modality stays down, LEAD the user-facing report with "coverage limited to X; systems documented only in Y may be missing" instead of burying it, and ask the user for known instances as a recall checklist. (Cost: missed **T-Run** in the Yandex agent-systems survey — user had to correct "плохо поисследовал", 2026-07-02.) |

## Optional session metric

After long sessions (>10 tool calls), one line for the user:

`Delegation: Task=N; root Edit/Write=M.`

Goal: lower M on ticket-sized tasks.
