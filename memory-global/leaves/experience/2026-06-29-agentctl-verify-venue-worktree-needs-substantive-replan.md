# agentctl verifies a stage in repo_root, not the worktree it ran in; only a substantive replan can move the venue

## Difficulty
Executing a substantive agentctl plan in an isolated git worktree (chosen so a parallel session's dirty tree is not entangled) silently diverges the verification venue from the engine's bookkeeping. The engine stores meta.repo_root once at submit-plan and runs every stage's verify_command as 'cd <repo_root> && <cmd>' — i.e. against the MAIN checkout, which lacks the worktree's commits — so each genuinely-green stage is recorded FAILED and routed to DIAGNOSING. Worse, the obvious fix (edit the plan's verify_command/repo_root to the worktree and replan) does NOT take: diff_plans classifies a verify_command/repo_root-only change as a 'refinement', and the refinement branch of cmd_replan reassigns only title/result/means/method/conditions/invariants — never verify_command or repo_root. Those two fields are reloaded ONLY by a 'substantive' replan (state.stages=new.stages; state.repo_root=new.meta.repo_root), and the structural signature that triggers substantive excludes verify_command — so a structural field (e.g. a stage done_criterion) must be changed to force it.

## Order & criterion
Make the engine verify in the venue that holds the commits. Either (a) keep repo_root = main checkout and accept the double-cd (verify_command itself 'cd <worktree> && ...' wins over the engine's prepended 'cd <repo_root> &&'), OR — when stages were already loaded with main-checkout verify_commands — force a SUBSTANTIVE replan: bump a structural field (a stage's done_criterion) so diff_plans returns 'substantive', which reloads state.stages (worktree verify_commands) + state.repo_root=worktree, returning to PLAN_READY for re-approval. Criterion: state.json shows repo_root=worktree and every stage verify_command points at the worktree; record-result then runs the real check green in-venue.

**Acceptance check:** agentctl status + state.json: repo_root and all stage verify_commands resolve to the worktree; each stage record-result passes its real verify_command (no rubber-stamp); verify-final green.

### 2026-06-30 — same reload gap, NON-worktree, and the destructive-substantive-replan trap
- Where it arose: DEEPAGENT/global: 7-stage `junk/the0/agents/` workspace-restructure plan (projects/ + common/ split). NOT a worktree this time — the verify_command divergence surfaced from a plain bug: stage 6's `verify_command` used `awk 'NR>140'` (counts lines across the *concatenation* of all files) where it meant `awk 'FNR>140'` (per-file line number). The engine auto-ran the stale NR command at record-result and false-FAILED the genuinely-green stage. Same class as the worktree case: **the field the engine actually executes (`verify_command`) cannot be corrected by a refinement replan**, because `_structural_signature` (plan.py) excludes verify_command, so a check-only fix always classifies as refinement, and the refinement branch of `cmd_replan` never copies verify_command into state.
- **KEY new tradeoff — a substantive replan RESETS already-PASSED stages.** The "force substantive" remedy from the initial context is *destructive* when earlier stages are already verified: the substantive branch does `state.stages = new.stages`, which re-initialises every stage outcome to pending — throwing away stages 1–5's PASSED records to fix a single field in stage 6. The cheaper, surgical remedy: **hand-patch the engine state store directly.** The state store (`~/.claude/agentctl/state/<session>.json`, a `FileStateStore`) is plain JSON and is NOT covered by the production gate (the gate guards settings/skills/agents/CLAUDE.md/instruction-repo/plans — not runtime state). Patch only `stages[k].criterion.verify_command` (NR→FNR), back the file up first, leave the 5 PASSED outcomes intact. Then a replan that targets the *node* (DIAGNOSING→VERIFYING) + re-arms the one stage suffices — no stage reset.
- **Registered-plan in-place-edit trap.** `cmd_replan` does `old = _load(state.plan_path)` and the refinement branch sets `state.plan_path = args.plan`. So after a refinement, `state.plan_path` *is* the file you last submitted — editing THAT file in place to build the next correction makes `old == new`, and the coverage gate's CHANGE check (the new stages' `(means, method)` multiset must differ from old) can then never pass. Remedy: submit each correction round as a **fresh, separately-named file** (plan-replan-v2.toml, -v3.toml, …), never by mutating the file at `state.plan_path`.
- **Flaky auto-dispatch on already-done work + no prior-dispatch guard.** `dispatch` for a `spawn:developer` stage auto-spawns a verifier synchronously; on work that is already complete this is non-deterministic (observed: one run COMPLETED at $0.36, an identical re-run MALFORMED→BLOCKED because the spawned developer misread the dispatch as "execute the whole plan"). `record-result` has **no** guard requiring a fresh dispatch — it needs node=EXECUTING + a non-None active stage + a non-empty `--control` attestation. Reliable path on already-done work: `unblock` (clear the BLOCKED) → `record-result --status passed --control "<genuine review attestation>"`, skipping dispatch entirely.
- Cross-ref: [[2026-06-24-developer-marker-not-on-line-1-false-block]] (the auto-spawned developer's MALFORMED marker is the same false-block failure mode).

### 2026-06-30 — THIRD instance (verify_command typo) + the surgical remedy was missed again
- Where it arose: DEEPAGENT/global: the `principle-rule-vs-perception` self-improvement plan (reframe the CLAUDE.md core principle). The engine auto-ran it at record-result and false-FAILED a genuinely-green stage (commit done, verify-all 14/14). Identical class to the two contexts above.
- **The recurrence is the lesson.** This is the 3rd logged instance of "the field the engine executes (`verify_command`) cannot be corrected by a refinement replan." Two independent remedies are already documented here: (a) force-substantive (bump a structural field → `state.stages=new.stages`) — *destructive*, resets PASSED stages; (b) surgical hand-patch of `~/.claude/agentctl/state/<session>.json` `stages[k].criterion.verify_command` (state store is NOT gated) — cheap, non-destructive. **I again took the destructive path (a)** and re-ran stages 1–3 (cheap idempotent greps here, so tolerable) — but I did not recall this leaf before choosing, so I paid the heavier remedy a third time. Discipline fix: on ANY engine false-FAIL of a measurable stage, **read this leaf first** and prefer remedy (b).
- **Systemic signal (RESOLVED — engine fix landed):** three strikes across one leaf triggered a targeted engine change (committed 2026-06-30, task `agentctl-verify-command-replan-fix`). `diff_plans._prose` now includes `verify_command`/`expected_exit` per stage; `old.meta.repo_root != new.meta.repo_root` is added to the refinement trigger; the `cmd_replan` refinement branch now carries `criterion.verify_command`, `criterion.expected_exit`, and `state.repo_root` into live state. A verify_command-only correction is now a cheap, non-destructive refinement — PASSED stages are preserved. Remedies (a) (force-substantive) and (b) (hand-patch state.json) are no longer needed for this class of error.

### 2026-06-30 — FOURTH instance, DURING the fix: the engine fix removes the *correction cost*, not the *authoring antipattern*
- Where it arose: the `agentctl-verify-command-replan-fix` task itself (the engine change above). Stage 1's plan `verify_command` was again malformed at authoring — `python3 -m pytest scripts/tests/test_replan.py -q 2>&1 | grep -qiE 'verify|repo_root|carries'`: `pytest -q` prints only progress dots + an `N passed` summary, **never test-function names**, so the grep matched nothing and exited 1 → the engine false-FAILED a stage whose tests were 899-green. A 4th instance of the same false-FAIL class, committed *while shipping the fix for that very class*.
- **Two distinct difficulties, only one removed by the engine fix.** (1) *Correction cost* — once a verify_command is wrong, how expensive is the fix? The engine change removed this: the correction was a single verify_command-only **refinement** replan that carried the new command into live state with **no stage reset and no re-approval** (live-validated end-to-end here — `replan` returned `kind=refinement`, stage 1 went FAILED→PENDING, re-recorded green). Remedies (a)/(b) were not needed. (2) *Authoring antipattern* — writing a check whose observable does not contain what it greps for. The engine fix does **NOT** remove this; it remains operator cognition.
- **Residual discipline (the durable lesson):** author a `verify_command` against the **actual observable** the command emits, not an assumed one. Two concrete traps now seen: a tool's CLI signature (instance 3: `lint-prose-length.py FILE` when the script takes no positional), and a tool's *output format* (instance 4: greping test-function names out of `pytest -q`, which suppresses them). Before trusting a check, **run it once by hand and read its real output**; for "is test X present" prefer greping the **test file** (`grep -q 'def test_…' file`) or `pytest --collect-only -q`, never the green-run summary. The engine now makes the fix cheap, so the cost of this mistake is low — but a one-line manual dry-run of the check still avoids the whole difficulty cycle.

### 2026-06-30 — SIXTH instance: the SAME reload gap exists for `final_check`, and the 2026-06-30 engine fix did NOT cover it
- Where it arose: DEEPAGENT/global: the `retire-last-accessed-stamp` task (retire the last_accessed temporal-frontmatter mechanism). 5-stage plan, all stages already PASSED and both sub-PRs landed; the ONLY blocker to resolution was a defective plan-level `[[final_check]]` (FC4). FC4 shelled `python3 scripts/memory_dates.py x.md` to assert the validator rejects a re-introduced `last_accessed` — but `memory_dates.py` is an **importable module with no `__main__`/CLI**, so the invocation no-ops to exit 0, and FC4's `! …` flipped it to exit 1: the check could NEVER pass regardless of the (correct, landed) deliverable. `verify-final` re-runs every `[[final_check]]`, so it refused the RESOLUTION transition and stayed at VERIFYING.
- **The gap is `final_check`, NOT `verify_command` — and the landed 2026-06-30 fix is field-specific.** The earlier contexts' engine fix added per-stage `verify_command`/`expected_exit` to `diff_plans._prose` and made the `cmd_replan` refinement branch carry them into state. But **`final_check` was left out of BOTH**: `diff_plans` (plan.py) references `final_check` nowhere (neither `_structural_signature` nor `_prose`), so a final_check-only edit classifies as `no_change`; and the `cmd_replan` refinement branch (cli.py) updates per-stage `criterion.verify_command` but never `state.final_check` — only the **substantive** branch does (`state.final_check = new.meta.final_check`). So the cheap refinement path that now exists for stage verify_commands does NOT exist for final_checks: a buggy final_check is reloadable ONLY by a substantive replan (or remedy (b)).
- **I took the DESTRUCTIVE force-substantive path (a) again — the leaf's own remedy (b) was the right move and I missed it a fourth time.** I wrote a corrected plan to a fresh sibling file (FC4 → direct `validate_temporal` import-and-call; plus a meta.done_criterion bump to force `substantive`), ran a substantive replan → PLAN_READY → re-approved against standing approval (check-correctness fix, scope/deliverable byte-identical and already landed) → re-drove all 5 stages via next-stage+record-result (dispatch skipped to avoid re-spawns; one stage-1 dispatch fired a $0.46 verifier before I switched to direct record-result) → verify-final green. **Cheaper remedy (b), unchanged from the prior contexts:** hand-patch `~/.claude/agentctl/state/<session>.json` `final_check[k].command` directly (the state store is plain JSON, NOT covered by the production gate), back the file up first, then re-run `verify-final` — NO replan, NO stage reset, NO re-approval. For a final_check-only fix this is strictly better than force-substantive (which re-initialises all stage outcomes). Discipline fix reaffirmed: on ANY engine false-block rooted in a field the engine executes (`verify_command` OR `final_check`), read this leaf and prefer the state.json hand-patch (b).
- **Authoring antipattern (new vector — INTERFACE mismatch):** the check shelled an importable module as if it had a CLI. Like instances 3/4/5 (signature / output-format / scope-CWD), the durable fix is the same: author the check against the **actual interface** of what it calls — a module with no `__main__` must be exercised by `python3 -c "import m; …"`, not `python3 m.py ARGS` (which silently does nothing). A one-line manual dry-run of FC4 would have exposed it at authoring time.
- **Systemic signal (UNRESOLVED — engine gap to file):** the 2026-06-30 engine fix closed the refinement-reload gap for stage `verify_command` but left the structurally-identical gap open for `final_check`. A clean Core fix mirrors that change: add `final_check` (command/expected_exit list) to `diff_plans._prose`, and carry `state.final_check = new.meta.final_check` into the `cmd_replan` refinement branch — then a final_check-only correction becomes a cheap, non-destructive refinement, same as verify_command is today. Filed as a Core difficulty (not fixed mid-resolution: agentctl/plan.py + cli.py are protected-Core, needing their own approval spine).

## Common core & variations
**Common:** Same root as all five prior contexts: the engine runs a literal check (per-stage `verify_command` at record-result, or every `[[final_check]]` at verify-final) so an over-broad / wrong-venue / wrong-interface check false-FAILS a genuinely-green stage or refuses resolution. The durable discipline is unchanged and now six-times-confirmed: author each check against the ACTUAL observable+venue+interface, and run it by hand once before trusting it. The two engine-correction remedies generalise across BOTH executed fields: (b) hand-patch `state.json` (`stages[k].criterion.verify_command` OR `final_check[k].command`) is cheap+non-destructive and always available; (a) force-substantive resets PASSED stages and should be the last resort. The 2026-06-30 refinement-reload engine fix covers stage `verify_command` only — `final_check` still needs (b) or substantive (instance 6). Scope an absence/purity check to the stage's changed files (the task delta), not the repo; prepend an explicit `cd <venue>` whenever the check touches a repo other than [meta] repo_root.

**Variations:** Unlike instances 3/4 (CLI-signature and output-format typos), the false-fail vectors here are SCOPE (recursive scan reaching unrelated files) and CWD (check venue != repo_root). And the correction cost was now cheap, validating the 2026-06-30 engine fix end-to-end on a real task: the stage-3 fix went in as a single verify_command-only REFINEMENT replan (reloaded the whole revised TOML via `replan --plan <revised>`; engine returned kind=refinement and carried the scoped check into live state with NO stage reset, no re-approval — the already-landed Core commit 5d9ba8e stayed PASSED). Force-substantive (a) and hand-patch state.json (b) were not needed. Net: the engine fix removed the correction cost for the SCOPE/CWD vectors too, but the authoring antipattern (a check that does not match its real observable OR venue) remains operator cognition — pre-empted only by the manual dry-run.

## Cost
High for the diagnosis, low for the fix. The 5-stage plan itself was small mechanical edits; the cost concentrated in two false-FAIL difficulty cycles on stage 2 before the root cause (refinement replan does not reload `verify_command`/`repo_root` — only a substantive replan does) was isolated. Net overhead vs. a clean run: ~2 extra difficulty cycles + one full substantive replan (v2→v3) + reading `cmd_replan`/`gates.py`/`plan.py` to localize the reload gap. No spawned specialists (all stages driven in-thread via next-stage+record-result, dispatch deliberately skipped to avoid re-spawning already-done work).
